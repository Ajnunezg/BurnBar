# OpenBurnBar SOTA Remediation Plan
This plan synthesizes the three diligence reviews into one execution roadmap. It treats live repo evidence as authoritative when reviews conflict.
## Problem Statement
OpenBurnBar has a serious startup-grade foundation, but it is not yet SOTA or Series A-clean because release truth, security proof, test discipline, app-layer concurrency, and architectural boundaries are inconsistent. The strongest parts of the system are the daemon contracts, local-first posture, release automation, and documentation. The weakest parts are live CI failure, a syntactically broken tag script, version drift, encryption-key recovery semantics, skipped/stale tests, red CodeQL, app-layer `@MainActor` plus `Task.detached` patterns, and partially extracted service spines.
## Current State
The branch is clean at `b1db1412` and even with `origin/release/openburnbar-0.1.2-beta.12`. Live GitHub evidence shows `OpenBurnBar PR Harness` failed and `CodeQL` failed on that same SHA. The release tag script is currently invalid shell syntax: `scripts/tag-release.sh:124` reports an unmatched single quote and `scripts/tag-release.sh:168` reports unexpected end of file.
The product truth is split across versions: `README.md:8` says `0.1.2-beta`, `project.yml:16` says `0.1.3-beta.1`, `CHANGELOG.md:73` has `0.1.3-beta.1`, `extensions/openburnbar/package.json:5` says `0.1.2-beta`, `homebrew/burnbar.rb:14` says `0.1.2-beta`, and `OpenBurnBarDaemon/Sources/OpenBurnBarDaemon/OpenBurnBarDaemonConfiguration.swift:65` says `0.1.2-beta`.
Security posture is better than prototype-grade but not diligence-clean. The threat model documents optional SQLCipher and a deterministic recovery file at `docs/THREAT_MODEL.md (129-145)`. The implementation writes a plaintext key recovery file with `0o600` permissions in `AgentLens/Services/DataStore/DatabaseEncryptionService.swift (123-165)`. That protects against casual disclosure but weakens SQLCipher’s claim against same-user local compromise.
The daemon gateway has strong local defaults and request-size limits, but it uses raw `NWParameters.tcp` in `OpenBurnBarDaemon/Sources/OpenBurnBarDaemon/OpenBurnBarHTTPGatewayServer.swift (45-55)`. Non-loopback modes need either enforced loopback-only launch posture or TLS/mTLS before they can be called SOTA.
The review claim that MissionControl has zero tests is stale for this checkout: `OpenBurnBarDaemon/Tests/OpenBurnBarDaemonTests/OpenBurnBarMissionControlServiceTests.swift` exists and is 5,605 lines. The real testing concern is not zero MissionControl coverage; it is active skipped tests, quarantined suites, red CI, missing mobile/functions/cloud proof gates, and insufficient evidence that performance/visual/security regressions block landing.
## SOTA Definition
SOTA for this repo means every launch artifact is reproducible, signed, tested, and version-consistent; every critical trust boundary has executable proof; every primary surface has green CI gates; cloud and mobile claims are either fully tested or explicitly out of launch scope; local-first data paths degrade safely; security docs match code; and architecture boundaries accelerate the next 12-24 months instead of hiding partial extractions.
The target verdict after this plan is “Strong startup-grade foundation” as the minimum acceptable bar, with a credible path to “Series A-quality technical foundation.”
## Remediation Strategy
Work in four ordered waves. Wave 0 restores truth and ship safety. Wave 1 closes launch-blocking security and CI proof gaps. Wave 2 hardens architecture, performance, and reliability. Wave 3 turns the repo into diligence-grade SOTA by adding measurable quality budgets and removing ambiguity from docs, tests, and release operations.
## Wave 0: Restore Release Truth and Green Baseline
The first objective is to make the current branch objectively safe to build and tag.
Fix the failing `OpenBurnBar PR Harness` app-test job and the failing `CodeQL` run before touching broader refactors. Pull the failing logs for run `25297234878` and CodeQL run `25297234880`, reproduce locally where possible, and land the smallest fixes that return those gates to green.
Fix `scripts/tag-release.sh` so `bash -n scripts/tag-release.sh` passes, then add script syntax validation to CI for all release and test scripts. The bug is likely in the `sed` expression around `scripts/tag-release.sh:92`, where the single quote in `s/'//g` breaks single-quoted shell parsing.
Collapse version drift into a single source of truth. Pick the intended release line, likely `0.1.3-beta.1` based on `project.yml:16` and `CHANGELOG.md:73`, then update README, extension package metadata, daemon version, Homebrew cask, docs, and release notes together. Add a CI verifier that fails when these surfaces diverge.
Re-run and require these baseline validators before moving to Wave 1: `./scripts/test-openburnbar-swift.sh`, `./scripts/test-openburnbar-app.sh`, `./scripts/test-openburnbar-ts.sh`, `./scripts/test-openburnbar-retrieval-evals.sh`, `./scripts/test-openburnbar-replay-evals.sh`, `./scripts/test-openburnbar-extension-host.sh`, `npm --prefix functions run lint`, `npm --prefix functions run build`, `bash -n scripts/*.sh`, and the GitHub PR harness.
## Wave 1: Close Launch-Blocking Security and Proof Gaps
The second objective is to make security claims executable, not aspirational.
Redesign database key recovery. The current recovery file in `AgentLens/Services/DataStore/DatabaseEncryptionService.swift (123-165)` should not be a plaintext copy of the SQLCipher key. Acceptable SOTA options are user-passphrase-wrapped recovery, Keychain-only with explicit data-loss tradeoff, Secure Enclave-backed wrapping where available, or an exportable recovery bundle that requires user action. Update `docs/THREAT_MODEL.md (129-145)`, the settings UI, recovery tests, and runbooks to match the chosen model.
Harden the HTTP gateway. If gateway mode is product-scoped only for localhost, enforce loopback-only regardless of config and make non-loopback impossible. If non-loopback is a supported mode, add TLS or mTLS to `OpenBurnBarDaemon/Sources/OpenBurnBarDaemon/OpenBurnBarHTTPGatewayServer.swift (45-55)`, rotate gateway bearer tokens, and add integration tests for plaintext-token rejection, host validation, CORS behavior, rate limiting, and oversized requests.
Make App Check and Firestore security verifiable. `docs/THREAT_MODEL.md (154-164)` depends on console-side App Check enforcement. Add repo-local proof: Firestore rules emulator tests, Cloud Functions auth/App Check unit tests, and a release checklist artifact that records enforcement state. If these cannot be automated, the docs must downgrade the launch claim from “enforced” to “operator-verified external setting.”
Sanitize telemetry. `AgentLens/Services/AppLogger.swift (60-88)` masks metadata in OSLog but sends raw metadata dictionaries into Sentry breadcrumbs. Add a sanitizer that redacts paths, tokens, auth headers, project names, model prompts, and user content before Sentry receives metadata. Back this with tests that assert sensitive keys and values are not emitted.
Eliminate critical silent data loss. Replace `try? insertUsages` and other write-path `try?` cases with logged, surfaced, retryable failures. Keep non-critical best-effort failures quiet only through `AppLogger.silentFailure`, and ensure those breadcrumbs are sanitized first.
## Wave 2: Make Testing, CI, and Delivery Diligence-Clean
The third objective is to make the repo’s quality gates match the breadth of the product.
Triage all active skips. There are 30 skip occurrences across test surfaces. Each one should become either an active passing test, a quarantined test with a reason and owner, or a deleted obsolete test. Active suites should not contain stale skips without issue references and expiry dates.
Revive or close quarantined tests. `AgentLensTests/README.md (3-29)` allows quarantine, but SOTA requires governance. Add a quarantine manifest with reason, owner, source subsystem, revival criteria, and target date. Move parser/performance tests back to `Active` when they compile against current contracts, or replace them with equivalent active coverage.
Expand CI to match launch scope. If mobile, functions, Firestore rules, cloud sync, and release smoke are part of launch claims, they need required gates. Add mobile tests, Functions lint/build/test/audit, Firestore rules emulator tests, shell syntax validation, CodeQL, actionlint, SwiftPM lockfile checks, retrieval/replay evals, extension-host tests, and authenticated daemon health smoke to required CI.
Make coverage actionable. The PR harness already extracts total and diff coverage in `.github/workflows/openburnbar-pr-harness.yml (63-112)`. Promote this into a persistent trend artifact with module-level thresholds. Keep diff coverage at 80% and ratchet total coverage by subsystem rather than chasing one global number.
Add visual and performance gates. SnapshotTesting is already a dependency in `project.yml:45-47`. Use it for the highest-risk UI surfaces, especially dashboard, onboarding, provider routing cockpit, chat panel, and settings. Add performance baselines for parser sweeps, projection pipeline, HNSW search, exact fallback, SQLite hydration, and launch/startup path. Fail CI on large regressions, not just functional failures.
## Wave 3: Harden Architecture and Runtime Behavior
The fourth objective is to reduce rewrite risk and make the architecture scale with product ambition.
Decompose daemon RPC dispatch. `OpenBurnBarDaemon/Sources/OpenBurnBarDaemon/OpenBurnBarDaemonServer.swift (287-405)` starts a long switch that centralizes authorization, decoding, logging, dispatch, and response encoding. Replace it with protocol-based handlers or a registry that preserves typed request/response contracts while adding middleware for auth, rate limiting, logging, metrics, and error mapping.
Finish partial service extractions. `project.yml (64-73)` excludes local split drafts for projection, search, and chat UI because they duplicate compiled root files. SOTA requires either completing those refactors or deleting the drafts. The repository should not carry “future architecture” files excluded from builds.
Move app I/O and indexing off `@MainActor`. The reviews consistently flag `Task.detached` and main-actor service work. Convert long-running parsing, sync, projection, search, iCloud mirroring, CLI bridge, and provider quota polling to actors or explicitly isolated services. Preserve UI updates on the main actor, but move disk, network, database, and vector work off it.
Set memory and correctness guardrails for vector search. The custom HNSW work can remain if it is proven. Add memory-budget enforcement, filtered-search behavior, delete/update tests, recall benchmarking against exact search, corruption handling, persistence compatibility tests, and an escape hatch to lexical-only retrieval. Document why custom HNSW beats a proven library for this product.
Make cloud/mobile architecture honest. If mobile remains cloud-first, docs should say mobile is cloud-companion rather than local-first. If local-first is the SOTA goal, add iOS local persistence, offline reads, sync replay, and conflict handling. Either path is acceptable; ambiguity is not.
## Wave 4: SOTA Polish and Investor-Grade Evidence
The final objective is to make the repo legible and impressive to a CTO, investor diligence team, or excellent engineer.
Create a launch-readiness dashboard generated by CI. It should include latest commit, required workflow status, coverage trend, skipped/quarantined test count, vulnerability count, CodeQL status, release script syntax status, version-consistency status, notarization smoke status, and App Check/Firestore proof status.
Create a security evidence packet. Include threat model, Firestore rules test results, Functions auth tests, gateway tests, key-recovery design decision, dependency supply-chain review, CodeQL status, secret-scanning results, and a short residual-risk register.
Create an architecture evidence packet. Include daemon RPC architecture, local-first data ownership, sync conflict model, retrieval/vector index design, mobile scope decision, and explicit “what is experimental vs core” boundaries. Keep it short, current, and linked from README.
Create a performance evidence packet. Include baseline data sizes, expected user envelopes, HNSW recall/latency, parser throughput, launch time, memory ceiling, Firestore read/write cost model, and degradation behavior.
Remove stale public-facing contradictions. README status, changelog, release docs, Homebrew cask, extension package, daemon version, and architecture docs should tell one story. Experimental surfaces should be labeled experimental; core surfaces should have green proof.
## Acceptance Criteria
The plan is complete when the current branch has green required GitHub workflows on HEAD, `scripts/tag-release.sh` passes syntax and dry-run validation, all version surfaces agree, CodeQL is green or explicitly waived with documented false positives, critical security proof is executable, active skips are governed, quarantined tests have owners and exit criteria, release smoke validates authenticated daemon RPC, app and Swift package tests pass locally and in CI, cloud/mobile/function claims are either gated or scoped out, Sentry metadata is sanitized, database key recovery no longer undermines encryption, and the architecture docs match the implemented system.
SOTA status requires more than green builds. The repo must also be able to answer diligence questions with evidence: what is tested, what is encrypted, what is local-first, what is experimental, what fails safely, what regresses visibly, and what will not need a rewrite in 12 months.
## Execution Order
Start with Wave 0 and do not begin broad refactors until HEAD is green and releasable. After Wave 0, security proof in Wave 1 should precede performance and architecture work because security gaps are launch-blocking. Wave 2 should run in parallel with Wave 3 only after the test harness can catch regressions. Wave 4 is the final evidence and documentation pass after implementation hardening lands.
